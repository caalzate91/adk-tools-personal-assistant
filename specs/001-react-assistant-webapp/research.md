**Status**: Complete — all NEEDS CLARIFICATION resolved

## Decision 1: ADK Web Server API Integration

**Decision**: Integrate via the native browser `EventSource` API against the ADK
`POST /run_sse` endpoint for streaming, and `POST /apps/{appName}/users/{userId}/sessions`
to create sessions. ADK defaults to `http://localhost:8000`.

**Rationale**: The ADK web server (`adk web`, FastAPI) exposes two chat endpoints:
- `POST /run` — synchronous, returns `list[Event]` when the full run completes.
- `POST /run_sse` — streaming SSE (`text/event-stream`), emits `data: <Event JSON>\n\n`
  for each agent step. This satisfies Constitution Principle IV (streaming MUST be used).

Reading from the ADK source (`adk_web_server.py`), the request body for both is:
```
RunAgentRequest {
  app_name:     string          // e.g. "personal_assistant"
  user_id:      string          // Firebase Auth uid
  session_id:   string          // UUID generated by POST /sessions
  new_message:  Content         // { role: "user", parts: [{ text: "..." }] }
  streaming:    boolean = false // set true for run_sse to get token streaming
  invocation_id?: string        // optional trace ID
}
```

Session lifecycle:
1. On app load (or after login): `POST /apps/personal_assistant/users/{uid}/sessions`
   → returns `{ id: sessionId, ... }`. Store `sessionId` in component state (not Firestore).
2. Each message turn: `POST /run_sse` with `{ streaming: true }` using `EventSource`.
3. On new conversation: delete old session with
   `DELETE /apps/personal_assistant/users/{uid}/sessions/{sessionId}`, create a new one.

**Alternatives considered**:
- `fetch()` with `POST /run` (synchronous): Rejected — loads full response before displaying,
  blocks UI, violates Constitution Principle IV.
- WebSocket `/run_live`: Rejected — designed for real-time voice/live inputs; adds unnecessary
  bidirectional complexity for a simple text chat.

## Decision 2: Firebase Authentication Provider

**Decision**: Google Sign-In (OAuth 2.0 popup) as the sole authentication provider.
Email/password is NOT included in this feature to keep scope minimal.

**Rationale**: "Firebase Auth Login" with Google is:
1. Zero friction for personal use — no password to manage.
2. Single-click from the Firebase Console — enable Google provider, add OAuth credentials.
3. Firebase SDK provides `signInWithPopup(provider)` — three TS lines.
4. The Firebase Auth `user.uid` maps directly to the ADK `user_id` field, creating a
   natural link between Firebase identity and ADK session ownership.

**Security note**: Firebase ID tokens are scoped per user. Firestore security rules
(see `contracts/firebase.md`) restrict reads/writes to `request.auth.uid == userId`,
preventing cross-user data access.

**Alternatives considered**:
- Email/password: Rejected — adds form complexity, password-reset flow, and email
  verification, all out of scope for a personal assistant.
- Anonymous auth: Rejected — data would be orphaned on token expiry; no cross-device
  access.
- Magic link (email link): Rejected — same inbox-friction as password without benefit.

## Decision 3: Firestore Conversation Storage Schema

**Decision**: Hierarchical path `users/{uid}/conversations/{convId}/messages/{msgId}`.
A conversation maps 1-to-1 with an ADK session. Messages are ordered by `timestamp`.

**Rationale**:
- Firebase security rules can be scoped at `/users/{uid}/**` — one rule covers all
  sub-collections.
- Conversation-level metadata (title, ADK session ID, timestamps) is stored on the
  conversation document; messages are a sub-collection for efficient pagination.
- Firestore real-time listeners (`onSnapshot`) on the messages sub-collection give live
  updates without polling — useful if multiple browser tabs are open.

Full schema in `data-model.md`.

**Alternatives considered**:
- Flat `conversations/{uid_convId}/messages/{msgId}`: Rejected — cannot scope security
  rules to the uid portion of a compound document ID.
- Single document with messages as array: Rejected — Firestore document size limit is
  1 MB; a long conversation could hit this limit.

## Decision 4: CSS Approach (Vanilla, No Preprocessors)

**Decision**: Global CSS custom properties (variables) for the design token layer; CSS
modules (`ComponentName.css` co-located with each component) for component-scoped styles;
CSS Grid and Flexbox for all layout. Zero Tailwind, zero Sass, zero CSS-in-JS.

**Rationale**: The user requirement is "use vanilla HTML, CSS, and JavaScript as much as
possible." Vite supports CSS modules out of the box with zero configuration. Custom
properties (`--color-primary`, `--spacing-md`, etc.) defined in `index.css` act as a
lightweight design token system without any build-time dependency.

**Token categories** (defined in `src/index.css`):
- `--color-*`: palette (background, surface, primary, text, error, success)
- `--space-*`: spacing scale (xs, sm, md, lg, xl)
- `--font-*`: family, size scale, weight
- `--radius-*`: border-radius scale
- `--shadow-*`: elevation shadows
- `--transition-*`: duration + easing

**Alternatives considered**:
- Tailwind CSS: Rejected — a utility CSS framework is a "library" that violates the
  vanilla CSS requirement.
- CSS Modules only (no custom properties): Rejected — duplicates token values across
  component CSS files; no single-source-of-truth for the design system.
- Styled-components / Emotion: Rejected — CSS-in-JS is a library dependency.

## Decision 5: State Management

**Decision**: React built-in state only (`useState`, `useReducer`, `useContext`). No
external state management library (no Redux, Zustand, Jotai, Recoil).

**Rationale**: The app has a simple state topology: auth state (one user) + one active
conversation (ordered list of messages) + request-in-flight flag. All three are manageable
with a `useConversation` hook (useReducer) and a `useAuth` hook (useState + Firebase
`onAuthStateChanged`). A context provider wraps the app for auth state sharing.

**State shape** (see `data-model.md` for full types):
```ts
// AuthContext
{ user: FirebaseUser | null; loading: boolean }

// useConversation
{
  messages:    Message[];
  status:      'idle' | 'sending' | 'streaming' | 'error';
  sessionId:   string | null;
  convId:      string | null;
  errorMessage: string | null;
}
```

**Alternatives considered**:
- Zustand: Rejected — lightweight but still an extra dependency; not justified for two
  state slices.
- Redux Toolkit: Rejected — heavyweight; YAGNI for a single-conversation app.

## Decision 6: Atomic Design Component Hierarchy

**Decision**: Strict Atomic Design five-layer hierarchy. Each layer has a defined
responsibility boundary enforced by code review (SC-006).

| Layer | Responsibility | Examples |
|-------|----------------|---------|
| **Atoms** | Unstyled or minimally styled primitive HTML wrappers | `Button`, `TextInput`, `Avatar`, `LoadingDots`, `Icon` |
| **Molecules** | Composed atoms with a specific UI purpose, no data-fetching | `MessageBubble`, `MessageInputBar`, `ToolResultWidget`, `ErrorBanner` |
| **Organisms** | Self-contained UI sections that may accept complex props | `ConversationFeed`, `ChatComposer`, `AppHeader` |
| **Templates** | Layout skeleton — defines slot positions, no business data | `ChatTemplate`, `LoginTemplate` |
| **Pages** | Full page wiring — connects hooks + services to templates | `ChatPage`, `LoginPage` |

**Rules enforced by code review**:
1. Atoms MUST NOT import from other atoms, molecules, organisms, templates, or pages.
2. Molecules MAY import atoms only.
3. Organisms MAY import atoms and molecules.
4. Templates MUST NOT import hooks or services (layout only).
5. Pages are the only layer that imports hooks, services, and contexts.
6. `services/` and `hooks/` are NOT part of the Atomic hierarchy — they are
   cross-cutting infrastructure imported only by Pages (and Organisms where necessary).

**Alternatives considered**:
- Feature-folder structure (`features/chat/`, `features/auth/`): Rejected — the user
  explicitly requested Atomic Design.
- Flat `components/` folder: Rejected — violates the explicit specification requirement.
